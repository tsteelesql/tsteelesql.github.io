AgentLint: Evidence-Driven Code Review with Optional LLM Explanations

Alpha ¬∑ Personal Project

As AI-assisted code and data pipelines scale, teams lose visibility into where risk accumulates. AgentLint surfaces evidence-bound security, performance, and quality signals before issues ship.

  • Multi-agent review across Security, Performance, and Quality for Python and SQL
  • Three modes expose cost/latency tradeoffs: default (<1s/file), explain (2‚Äì5s), deep (10‚Äì30s)
  • Findings persist across runs with CLI, JSON export, and a Streamlit dashboard
View signals, evidence, and limits Hide signals, evidence, and limits

Signals & Instrumentation

  • LangGraph-orchestrated workflow runs Security ‚Üí Performance ‚Üí Quality per file
  • CLI commands for review, list, show, init, and dashboard; JSON output for integration
  • SQLAlchemy-backed persistence supports repeatable runs and historical inspection
  • Streamlit dashboard enables review outside the terminal
  • Security findings reference CWEs (e.g., CWE-89, CWE-798, CWE-327, CWE-22, CWE-78, CWE-95, CWE-502)

Evidence & Limits

  • Workloads: Python (.py) and SQL (.sql) files analyzed independently (no cross-file analysis)
  • Metrics: File path, line number, category, severity, rule_id, description, confidence score; mode speed as listed above
  • Known gaps: No cross-file analysis, data-flow tracking, or type inference; limited SQL dialect support; no FP/FN benchmarks or large-repo perf tests
  • Next validation step: Benchmark false positive/negative rates against known-vulnerable codebases

Additional Metadata

MIT-licensed Python package (3.10+) using LangChain/LangGraph with optional LLM providers (OpenAI, Anthropic, Ollama).

Conversational Data Analyst: Observable LLM-to-SQL Pipeline

Alpha ¬∑ Experimental

Natural-language interface for exploring SQLite databases with visibility into schema retrieval, generated SQL, safety outcomes, execution timing, and visualization selection.

  • Six-stage LangGraph pipeline: retrieve ‚Üí generate ‚Üí validate ‚Üí execute ‚Üí visualize ‚Üí respond
  • RAG schema retrieval (ChromaDB + sentence-transformers) + SQL generation via OpenAI or Ollama
  • SELECT-only validator blocks mutation ops, comments, and multi-statement queries
View signals, evidence & limits Hide signals, evidence & limits

Signals & Instrumentation

  • Observable signals per stage: retrieved schema context, generated SQL, validation outcomes
  • Execution timing and row counts surfaced alongside query results
  • Visualization type selection rationale based on data shape

Evidence & Limits

  • Workloads: Local exploration and prototyping; understanding LLM-to-SQL pipeline behavior; evaluating RAG-based schema retrieval; testing safety validation approaches
  • Metrics: Execution timing and row counts tracked per query; cold start includes schema store initialization (~30s first run)
  • Known gaps: No semantic validation of SQL correctness; no LLM API cost tracking; no per-stage latency profiling; no retrieval evaluation metrics; not tested under load; SQLite-only
  • Next validation step: Start with questions where you can verify expected SQL; inspect retrieved schema context; test edge cases (ambiguous columns, missing relationships, follow-ups); review blocked queries to map validator boundaries

Additional Metadata

Python package (v0.1.0) with LangGraph orchestration, RAG-based schema retrieval, and a Streamlit UI.

RAG Evaluation Framework

Alpha ¬∑ Early Development

Evaluation toolkit for understanding RAG system behavior‚Äîretrieval quality, generation signals, latency, token usage, and cost estimation‚Äîbefore issues surface in production.

  • Retrieval + generation metrics: Precision/Recall@K, NDCG, MRR; BLEU, ROUGE-L, semantic similarity, LLM-as-Judge
  • Performance signals: latency percentiles (p50/p95/p99), token usage, cost estimation
  • Adapter pattern + decorator-based tracking; SQLite persistence and A/B tests with HTML reports
View signals, evidence & limits Hide signals, evidence & limits

Signals & Instrumentation

  • Retrieval quality: Precision@K, Recall@K, NDCG@K, and MRR to surface where retrieval fails
  • Generation quality: BLEU/ROUGE-L, semantic similarity, and optional LLM-as-Judge scoring (non-deterministic)
  • Performance: latency percentiles (p50/p95/p99), token usage, and cost estimation for bottleneck/cost-driver visibility
  • Structured evaluation runs against JSONL datasets (queries, expected documents, reference answers)
  • A/B tests compare configurations and can generate an HTML comparison report

Evidence & Limits

  • Workloads: Single-turn RAG evaluation only; not hardened for production monitoring or high-volume continuous evaluation; not tested beyond ~1000 docs/query
  • Metrics: Retrieval (Precision/Recall@K, NDCG@K, MRR); generation (BLEU, ROUGE-L, semantic similarity, LLM-as-Judge); performance (latency p50/p95/p99, token usage, cost estimation)
  • Known gaps: No multi-turn conversation tracking; no streaming support; no automatic dataset generation; limited provider error handling; BLEU/ROUGE are English-centric
  • Next validation step: Not explicitly stated in the repo

Additional Metadata

Python evaluation toolkit for RAG systems with adapter pattern, decorator-based tracking, and A/B testing support.
